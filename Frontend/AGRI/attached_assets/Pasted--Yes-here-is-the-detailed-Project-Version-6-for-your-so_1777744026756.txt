V6 scope
Version 6 should be the first fully operational, fault-tolerant, dual-mode release: it keeps the original core of UDLRN, state-aware routing, satellite fusion, AI fraud scoring, evidence generation, and verdict automation, but adds stable deployment, offline/local fallback, stronger observability, production-grade workflows, and role-based portals so the system can keep functioning even when Docker, one infrastructure service, or a state API fails.

The base system vision already defines a 30-second claim verdict, a unified UDLRN-driven intake, state-specific fraud logic, a 47-feature ML pipeline, 3-model ensemble scoring, evidence package generation, and decision routing thresholds for approve, review, field visit, and reject; V6 should convert that from a concept-heavy architecture into an implementation-complete product release.

V6 objectives
V6 should target six outcomes:

Reliable deployment in two modes: Docker-first and local split-mode fallback.

End-to-end claim flow that works even with partial subsystem outages.

Strong admin control over states, thresholds, rules, and queues.

Full auditability and explainability for every verdict.

Faster debugging through health checks, logs, metrics, and replay tools.

Operational readiness for pilot rollout in Maharashtra, Karnataka, Rajasthan, Telangana, Punjab, and Uttar Pradesh, which are already central to the state-aware architecture in the concept document.

The concept paper already identifies fragmented state land systems, bribable human verification, lack of automated satellite checks, and long processing times as the main reasons PMFBY fraud persists, so V6 should explicitly solve these as system requirements rather than just pitch points.

Dual-run deployment
Your biggest practical requirement is that if Docker fails, the system should still start as separate backend and frontend processes but behave the same. V6 should formally include two supported run modes:

Mode A — Container mode
docker compose up --build

Full infra: Postgres, Redis, Kafka, Cassandra, MinIO, Spring Boot services, Python services, Angular apps.

Used for integrated testing, demo, staging, production-like setup.

Mode B — Split local mode
Spring Boot services run with application-local.yml.

Angular portals run with ng serve.

Python services run with uvicorn.

Postgres and Redis local services mandatory.

Kafka/Cassandra/MinIO become optional with compatibility fallbacks.

V6 fallback rules
If Kafka is unavailable:

write orchestration events into event_outbox in Postgres;

use scheduled poller to continue the pipeline.

If Cassandra is unavailable:

write audit events into audit_fallback in Postgres.

If MinIO is unavailable:

write generated artifacts to local file store with same metadata index.

If one state API is unavailable:

switch to cache-first mode and satellite-first risk screening.

If Docker orchestration fails:

one command starts backend and frontend separately.

This fallback philosophy aligns with the concept document’s own warning that state APIs can crash frequently and require async caching with 90-day TTL plus satellite-only fallback mode.

V6 workflow
The source document gives the core claim sequence in nine steps from claim ingestion to audit trail. V6 should preserve this flow and add resilience controls at each stage.

Detailed V6 flow
Farmer/CSC enters UDLRN.

UDLRN service resolves farmer, parcel, state, season.

Claims service creates draft claim and idempotency key.

State router selects state adapter.

Land verification runs, with mutation and land-use checks.

Bank and policy duplication checks run.

Satellite job is launched or served from recent cache.

Feature engineering computes geospatial, temporal, operator, financial, and policy features.

AI ensemble returns score, explanations, and confidence.

Rule engine applies hard rules and threshold profile.

Evidence package is generated.

Decision is issued:

approve,

review,

field visit,

reject.

Notifications are triggered.

Audit event stream is finalized.

Analytics aggregates update.

Degraded workflow
If state adapter fails:

continue with cached land record if fresh enough;

otherwise create “provisional satellite-first review.”

If satellite retrieval fails:

continue with state and behavioral checks;

force manual review if score confidence is low.

If event bus fails:

continue through outbox pattern.

If report generation fails:

decision can still proceed, but case marked “evidence regeneration pending.”
