SECTION 2 — UDLRN GENERATION SYSTEM
═══════════════════════════════════════════════════════════════

UDLRN is a 14-character alphanumeric unique land record number.

FORMAT:  SS-DDDD-PPPPPP-CC
  SS     = 2-digit State Code (29=Karnataka, 27=Maharashtra, 
           36=Telangana, 03=Punjab, 09=UP, 08=Rajasthan)
  DDDD   = 4-digit District Code (from state land API response)
  PPPPPP = 6-digit Plot Hash (SHA-256 of surveyNo+village+taluk, 
           take first 6 hex chars, uppercase)
  CC     = 2-digit Luhn checksum for validation

EXAMPLE: 29-0572-A3F8C1-07 (Karnataka, Bagalkot, specific plot)

UDLRN EMBEDDED DATA (stored in PostgreSQL table `udlrn_master`):
  - GPS coordinates (lat/lng centroid of plot polygon)
  - Plot boundary polygon (GeoJSON, stored as PostGIS geometry)
  - Survey number, village, taluk, district, state
  - Owner full name, co-owners array, tenancy status
  - Aadhaar-linked bank account (masked), IFSC, DBT status
  - Land area in hectares (from land API)
  - Land use type (agricultural / non-agricultural / forest / govt)
  - Current declared crop, sowing season
  - 10-year crop history (JSON array)
  - All past PMFBY claim IDs and outcomes
  - 10-year NDVI baseline archive (JSON: date → NDVI value)
  - Fraud score history (array of past scores)
  - Blacklist status (boolean + reason if blacklisted)
  - UDLRN creation timestamp, last updated timestamp

GENERATION TRIGGER: Farmer Registration API call
  Step 1: Accept farmer input (Aadhaar, state, district, taluk, 
          village, survey number)
  Step 2: Call state land API → get land record
  Step 3: Verify Aadhaar name matches RTC/7-12 owner name 
          (fuzzy match, Jaro-Winkler similarity > 0.85)
  Step 4: Generate UDLRN using format above
  Step 5: Call KGIS API → get GPS polygon for plot
  Step 6: Trigger async Celery job → fetch 10yr NDVI baseline 
          from GEE
  Step 7: Save complete UDLRN record to PostgreSQL
  Step 8: Return UDLRN to farmer via WhatsApp (Gupshup) + SMS

--- KARNATAKA ---
KGIS API Base: https://kgis.ksrsac.in/kgis/

KGIS Hierarchy Endpoints (GET):
  /webapi/GetDistricts          → all Karnataka districts + codes
  /webapi/GetTaluks?districtId= → taluks under district
  /webapi/GetHoblis?talukId=    → hoblis under taluk
  /webapi/GetVillages?hobliId=  → villages + village codes
  
  These give GPS-linked administrative codes used to query 
  the plot polygon from KGIS map layers.

Bhoomi RTC API: https://landrecords.karnataka.gov.in/Service1.aspx
  Method: POST (SOAP/REST hybrid)
  Input:  { districtId, talukId, hobliId, villageId, surveyNo }
  Output: { ownerName, coOwners[], landArea, declaredCrop, 
            landUseType, mutationDate, mutationReason, 
            rtcYear, pahaniDetails }

Karnataka Custom Fraud Checks:
  1. VAO Falsification: mutationDate within 30 days of claim → 
     FLAG(VAO_FALSIFICATION) → alert District Collector
  2. NDVI vs RTC contradiction: NDVI > 0.6 + RTC says damaged → 
     auto-reject + flag officer
  3. Non-ag reclassification: landUseType changed in 90 days → 
     freeze UDLRN + Tahsildar notification
  4. Area inflation: landArea(RTC) > landArea(satellite polygon) 
     by >15% → FLAG(AREA_INFLATION)

--- MAHARASHTRA ---
Mahabhumi 7/12 Portal: https://abhilekh.mahabhumi.gov.in
Digital Satbara: https://digitalsatbara.mahabhumi.gov.in

Satbara (7/12) Endpoint:
  POST /bhulekh/satbara
  Input:  { region, district, taluka, village, surveyNo }
  Output: { ownerName, area, cropSeason1, cropSeason2, 
            landType, irrigationSource, encumbrances, 
            otherRights, mutationEntries[] }

8A Khata Endpoint:
  POST /bhulekh/8a
  Input:  { district, taluka, village, khataNo }
  Output: { ownerName, allSurveyNos[], totalArea, activePolicies }

Maharashtra Custom Fraud Checks:
  1. CSC Bulk Pattern: same CSC operator > 50 claims/day → 
     auto-block CSC ID + FIR alert
  2. Multi-policy: same owner > 3 active policies via 8A → 
     flag OVER_INSURANCE
  3. Non-agricultural land: landType != 'agricultural' → 
     auto-reject immediately
  4. Pre-sowing NDVI: NDVI on declared sowing date < 0.1 → 
     no crop planted → auto-reject

SECTION 4 — SATELLITE & ML PIPELINE (PYTHON MICROSERVICE)
═══════════════════════════════════════════════════════════════

This runs as a separate Python FastAPI Docker container.
Spring Boot calls it via REST: POST /satellite/analyze

INPUT to Python service:
{
  "udlrn": "29-0572-A3F8C1-07",
  "polygon": { GeoJSON polygon of plot },
  "sowing_date": "2025-06-15",
  "claim_date": "2025-09-20",
  "state": "KA",
  "declared_crop": "paddy"
}

SATELLITE DATA FETCHED (Google Earth Engine):
  1. Sentinel-2 (optical, 10m, 5-day revisit) — FREE
     - NDVI timeline from sowing to claim date (every 5 days)
     - True color image at claim date (B4/B3/B2)
     - NDVI health map at claim date
     - Loss map: NDVI(sowing) minus NDVI(claim_date)
     - Cloud masking: QA60 band filter
  2. Sentinel-1 SAR (radar, 10m, 6-day revisit) — FREE
     - VV/VH backscatter for monsoon cloud penetration
     - Soil moisture proxy
     - Used when optical cloud cover > 60%
  3. Landsat 8/9 (30m, 16-day) — FREE
     - 10-year NDVI baseline per plot
     - Historical land use classification
  4. ISRO Resourcesat-2A (5.8m) — via ISRO NRSC API/MOU
     - High-resolution boundary detection for small plots
     - Use when plot area < 0.5 hectare

GEE CODE PATTERN (Python ee SDK):
  - Authenticate via service account JSON key
  - Define geometry from plot polygon GeoJSON
  - Filter ImageCollection by date range and geometry
  - Apply cloud mask function
  - Calculate NDVI = (NIR - RED) / (NIR + RED)
  - Reduce to mean NDVI per image date
  - Return time series as JSON array

ML FRAUD SCORING:
  47 features engineered per claim:
  - ndvi_at_sowing, ndvi_at_claim, ndvi_delta
  - ndvi_max_in_season, ndvi_min_in_season
  - ndvi_std_dev (volatility), ndvi_trend_slope
  - sar_vv_mean, sar_vh_mean, sar_backscatter_delta
  - area_satellite_ha, area_rtc_ha, area_delta_pct
  - crop_phenology_match (0/1 based on declared crop calendar)
  - land_use_code, mutation_recency_days
  - csc_operator_daily_claim_count
  - owner_active_policy_count
  - historical_ndvi_baseline_mean (10yr avg)
  - ndvi_vs_baseline_delta
  - cloud_cover_pct, sar_used_flag
  - district_fraud_rate_30d (rolling)
  - village_fraud_rate_30d (rolling)
  - claim_amount_vs_expected_ratio
  ... (all 47 features documented)

3 MODELS (ensemble, majority vote with weighted scoring):
  Model 1 — Crop Classifier (Random Forest, 200 trees):
    Input: NDVI timeline + phenology features
    Output: probability crop exists and matches declared type
  Model 2 — Anomaly Detector (Isolation Forest):
    Input: all 47 features
    Output: anomaly score (how different from genuine claims)
  Model 3 — Timeline Validator (XGBoost):
    Input: temporal NDVI sequence
    Output: probability of genuine crop loss event

FINAL FRAUD SCORE = weighted average:
  (RF score × 0.35) + (IF score × 0.30) + (XGB score × 0.35)
  Normalized to 0–100

OUTPUT from Python service:
{
  "udlrn": "...",
  "fraud_score": 73,
  "ndvi_sowing": 0.62,
  "ndvi_claim": 0.58,
  "ndvi_loss_map_url": "s3://ndvi-reports/...",
  "true_color_url": "s3://satellite-images/...",
  "evidence_pdf_url": "s3://pdf-evidence/...",
  "flags": ["VAO_FALSIFICATION", "AREA_INFLATION"],
  "recommendation": "MANDATORY_FIELD_VISIT",
  "confidence": 0.89,
  "model_versions": { "rf": "1.2", "if": "1.1", "xgb": "1.3" }
}

ROLE HIERARCHY (Spring Security roles, each inherits lower):
  SUPER_ADMIN
    → Can create/delete admin accounts for all states
    → Can view all claims, all fraud data, national heatmap
    → Can modify fraud scoring thresholds
    → Access: everything
  STATE_HEAD
    → All districts within assigned state
    → Can override District Officer decisions
    → Access: all claims + officers in own state

    → Handles review queue (score 31–60)

  FIELD_INSPECTOR
    → Handles assigned CCE field visits only
    → Can upload field visit report + photos

PROCESSING PIPELINE (all steps logged to Cassandra audit trail):

Step 1 — Ingestion (1 second):
  - Validate UDLRN exists and matches JWT
  - Check blacklist status
  - Check duplicate claim (same UDLRN + season)
  - Save claim with status: INGESTED

Step 2 — State Router (1 second):
  - Read state prefix from UDLRN
  - Route to KarnatakaApiService / MaharashtraApiService / etc.

Step 3 — Land Record Verification (2–5 seconds):
  - Fetch fresh RTC / 7-12 from state API 
    (or Redis cache if < 90 days old)
  - Verify: owner, area, land use type, mutation recency
  - Apply state-specific fraud checks
  - If hard-fail (non-ag land, blacklisted): 
    instant reject, skip satellite
  - Save status: LAND_VERIFIED

Step 4 — Satellite Job Dispatch (async, Kafka event):
  - Publish event: ClaimSatelliteJobRequested
  - Python Celery worker picks up from Kafka topic
  - Fetches GEE data (15–45 seconds async)
  - Stores results to MinIO
  - Publishes event: SatelliteJobCompleted
  - Spring Boot Kafka consumer receives result

Step 5 — Fraud Feature Engineering (2–5 seconds):
  - Combine land record data + satellite result
  - Engineer all 47 features
  - Save status: FEATURES_ENGINEERED

Step 6 — ML Scoring (1–3 seconds):
  - Call Python FastAPI ML endpoint
  - Get fraud score 0–100 + flags array
  - Save status: SCORED

Step 7 — Evidence Package (5–10 seconds):
  - Generate PDF: true-color image + NDVI map + loss map + 
    score breakdown + flags + recommendation
  - Upload to MinIO
  - Save PDF URL to claim record

Step 8 — Autonomous Verdict (1 second):
  SCORE 0–30   → AUTO_APPROVE
    - Update claim status: APPROVED
    - Trigger DBT payout initiation (PFMS API call)
    - Send WhatsApp: "✅ Claim #ID approved. ₹X will be 
      credited to XXXX1234 within 3 working days."
    
  SCORE 31–60  → OFFICER_REVIEW
    - Add to district officer review queue
    - Send WhatsApp to farmer: "🔄 Claim under review. 
      You will hear back within 5 working days."
    - Notify district officer via email + dashboard alert
    
  SCORE 61–80  → MANDATORY_CCE_VISIT
    - Assign to available Field Inspector in taluk
    - Send WhatsApp to farmer: "📋 Field visit scheduled. 
      Inspector will contact you within 3 days."
    - Notify Field Inspector with GPS coordinates of plot
    
  SCORE 81–100 → AUTO_REJECT + FIR_ALERT
    - Update claim status: REJECTED_FRAUD
    - Freeze UDLRN (no new claims for 180 days)
    - If CSC-filed: increment CSC daily counter, 
      auto-block if > 50/day
    - Send FIR alert to District Collector with 
      evidence PDF URL + satellite coordinates
    - Send WhatsApp to farmer: "❌ Claim rejected. 
      Reason: [flag description]. Appeal: /appeal/ID"
    - Log to Cassandra with immutable timestamp

Step 9 — Audit Trail (background, always):
  - Every step logged to Cassandra with:
    { claimId, udlrn, stepName, timestamp, actorId, 
      inputSnapshot, outputSnapshot, modelVersion, 
      satelliteImageUrls, decisionReason }
  - Court-admissible, cryptographically signed entries

IMPORTANT IMPLEMENTATION RULES
═══════════════════════════════════════════════════════════════

1. NEVER use hardcoded/mock data. Every API call must be real.
2. All passwords: bcrypt cost 12. Never log passwords.
3. Aadhaar numbers: store only SHA-256 hash. Never raw.
4. JWT secrets: from environment variables, never in code.
5. All DB connections: connection pool via HikariCP (Spring default)
6. State API failures: fail gracefully → satellite-only mode → 
   flag for manual review. Never crash the claim pipeline.
7. GEE rate limits: implement exponential backoff + Redis 
   cache for all satellite results (TTL 30 days per claim).
8. Audit log (Cassandra): APPEND ONLY. No updates, no deletes.
9. All images returned to frontend: presigned MinIO URLs 
   (valid 1 hour), never expose raw bucket.
10. TOTP for admins: use java-otp library (RFC 6238 compliant).
11. All API responses follow standard envelope:
    { "success": boolean, "data": {}, "error": null, 
      "timestamp": ISO8601, "requestId": UUID }
12. Farmer can NEVER see another farmer's data — enforce 
    UDLRN ownership check at service layer, not just JWT.

All topics auto-created. All consumers @RetryableTopic(attempts=3).

PIPELINE FLOW (each step is a separate Kafka consumer):

STEP 0: claim.pre-validation
  Triggered: on claim filed
  Check: is UDLRN frozen? → reject immediately
  Check: is farmer blacklisted? → reject immediately
  Check: udlrn_season_lock exists? → reject DUPLICATE_POLICY
  Check: CSC rate limit (Redis) if filed by CSC
  Check: cross-state duplicate (udlrn_season_lock)
  On pass: INSERT udlrn_season_lock + publish claim.filed

STEP 1: claim.filed → LandVerificationConsumer
  Fetch plot polygon from PostGIS (already stored at registration)
  Recalculate area: ST_Area(ST_Transform(plot_polygon, 32643))/10000
  Fetch fresh RTC from Bhoomi/Mahabhumi API
  Check: RTC mutation date < 30 days → VAO_FALSIFICATION flag
  Check: rtc_area vs kgis_area delta > 20% → AREA_INFLATION flag
  Check: land_use_type == Agricultural → else NON_AG_LAND flag
  Check: KGIS Nearby Location Details → Urban? → flag
  Store: rtc_raw JSON, mutation_days_before, area_delta_pct
  Publish: claim.land.verified

STEP 2: claim.land.verified → BankVerificationConsumer
  Call Protean Bank Account Verification (API 96)
  Call Protean Name Match (API 88): bank_name vs rtc_owner_name
  If match_score < 80: FLAG BANK_DBT_FRAUD
  Update: claims.bank_name_match_score, claims.bank_verified_at
  Publish: claim.bank.verified

STEP 3: claim.bank.verified → SatelliteDispatchConsumer
  POST to Python satellite service:
  { udlrn, polygon_wkt, sowing_date, claim_date, 
    damage_type, crop, district, season_type,
    expected_peak_ndvi (from crop_phenology_calendar),
    landsat_baseline_ndvi (from udlrn_master) }
  Celery queues job. Returns job_id immediately.
  Store: claims.pipeline_stage = 'SATELLITE_QUEUED'
  Publish: satellite.job.requested

STEP 4: satellite.job.completed → FraudScoringConsumer
  Receives full satellite result from Python.
  Runs ClaimScoringService.calculateFinalScore():
    base_score = satellite.fraud_score
    Apply all flag weights (see Section 6)
    Cap at 100, floor at 0
  Generates flag breakdown JSON
  Publishes: claim.scored

STEP 5: claim.scored → VerdictRouter
  score  0-30:  → claim.approved
  score 31-60:  → claim.review.queued
  score 61-80:  → claim.cce.assigned
  score 81-100: → claim.auto.rejected (+ FIR if >90)

  → kyc-service: initiate eSign workflow
  → dbt_payouts: insert record (DBT triggered)

STEP 6b: claim.review.queued

  → Cassandra audit

STEP 6c: claim.cce.assigned

  → Publish final verdict event

PART 7 — PYTHON SATELLITE SERVICE (satellite-service/main.py)
════════════════════════════════════════════════════════════════════════

FastAPI + Celery (Redis broker) + Google Earth Engine

ENDPOINT: POST /satellite/analyze
Input: SatelliteAnalysisRequest {
  udlrn, polygon_wkt, sowing_date, claim_date,
  damage_type, crop_type, district_name,
  season_type, expected_peak_ndvi,
  landsat_baseline_ndvi
}
Returns: job_id (Celery task)

ENDPOINT: GET /satellite/result/{job_id}
Returns job status + result when complete.

CELERY TASK: analyze_plot(request):

  STEP 1 — Parse + GEE geometry:
    geom = shapely.wkt.loads(polygon_wkt)
    ee_geom = ee.Geometry.Polygon(
        [list(geom.exterior.coords)])

  STEP 2 — Landsat 10-year baseline:
    landsat = ee.ImageCollection('LANDSAT/LC08/C02/T1_L2')
      .filterBounds(ee_geom)
      .filterDate('2013-01-01', '2023-12-31')
      .map(mask_clouds_landsat)
      .map(compute_ndvi_landsat)
    max_ndvi = landsat.select('NDVI').reduce(
        ee.Reducer.max())
    baseline_val = max_ndvi.reduceRegion(
        ee.Reducer.mean(), ee_geom, 30).getInfo()
    if baseline_val < 0.15:
        flags.append('PHANTOM_FARM')

  STEP 3 — Try Sentinel-2:
    s2 = ee.ImageCollection('COPERNICUS/S2_SR_HARMONIZED')
      .filterBounds(ee_geom)
      .filterDate(sowing_date - 7days, claim_date + 3days)
      .filter(ee.Filter.lt(
          'CLOUDY_PIXEL_PERCENTAGE', 20))
      .map(mask_s2_clouds)
      .map(compute_ndvi_s2)
    
    Get cloud cover on claim date:
    cloud_pct = get_cloud_cover(ee_geom, claim_date)
    
    if cloud_pct > 60:
        use_sar = True  → STEP 3b
    else:
        use_sar = False → compute NDVI

  STEP 3a — Sentinel-2 NDVI:
    ndvi_series = [] # array of {date, ndvi}
    for image in s2_collection:
        date = image.date()
        ndvi = image.normalizedDifference(['B8','B4'])
        val = ndvi.reduceRegion(
            ee.Reducer.mean(), ee_geom, 10).getInfo()
        ndvi_series.append({date, val})
    
    ndvi_sowing = interpolate_for_date(
        ndvi_series, sowing_date)
    ndvi_claim = interpolate_for_date(
        ndvi_series, claim_date)
    ndvi_loss_pct = (ndvi_sowing - ndvi_claim) / 
                     ndvi_sowing * 100 if ndvi_sowing > 0
    
    # Pre-sowing check
    ndvi_before_sowing = interpolate_for_date(
        ndvi_series, sowing_date - 14days)
    if ndvi_before_sowing > 0.5:
        flags.append('RETROACTIVE_CLAIM')
    
    # Phenology check
    if ndvi_claim < expected_peak_ndvi * 0.5 AND
       days_since_sowing < 60:
        flags.append('CROP_PHENOLOGY_MISMATCH')

  STEP 3b — Sentinel-1 SAR fallback:
    s1 = ee.ImageCollection('COPERNICUS/S1_GRD')
      .filterBounds(ee_geom)
      .filterDate(sowing_date, claim_date)
      .filter(ee.Filter.eq('instrumentMode','IW'))
      .select(['VV','VH'])
    
    vv_sowing = get_sar_value(ee_geom, sowing_date, 'VV')
    vv_claim = get_sar_value(ee_geom, claim_date, 'VV')
    vh_sowing = get_sar_value(ee_geom, sowing_date, 'VH')
    vh_claim = get_sar_value(ee_geom, claim_date, 'VH')
    
    vv_drop = vv_sowing - vv_claim  # in dB
    vh_drop = vh_sowing - vh_claim
    
    if vv_drop > 3: flood_confirmed = True
    # Convert SAR to pseudo-NDVI equivalent for scoring

  STEP 4 — IMD Weather:
    GET https://imdpune.gov.in/api/weather/district
    ?district={district_name}&date={damage_date}
    
    If damage_type = FLOOD:
        if imd_heavy_rainfall == False: 
            flags.append('WEATHER_MISMATCH')
            fraud_score_adj += 25
        else: 
            fraud_score_adj -= 20
            imd_confirmed = True
    
    Similar logic for DROUGHT (check IMD drought index),
    HAIL (check IMD hail alerts), 
    CYCLONE (check IMD cyclone track).

  STEP 5 — Export satellite images to MinIO:
    true_color = s2_latest.select(['B4','B3','B2'])
    ndvi_map = ndvi_image.visualize(
        min=0, max=1,
        palette=['red','yellow','lightgreen','green'])
    loss_map = ndvi_loss.visualize(
        min=-0.5, max=0.5,
        palette=['green','white','red'])
    
    Export each as PNG (300x300px) → MinIO bucket
    Generate NDVI timeline chart (matplotlib) → MinIO
    Generate SAR image if SAR used → MinIO

  STEP 6 — XGBoost + IsolationForest scoring:
    features = build_feature_vector(47 features)
    xgb_score = xgb_model.predict_proba(features)[1]*100
    iso_score = isolation_forest.score_samples(features)
    iso_normalized = normalize_isolation_score(iso_score)
    final = 0.6*xgb_score + 0.4*iso_normalized
    
    # Apply flag adjustments
    for flag in flags:
        final += FLAG_WEIGHTS[flag]
    final = max(0, min(100, final))

  STEP 7 — Return full result:
    {
      fraud_score, confidence, data_source,
      ndvi_sowing, ndvi_claim, ndvi_loss_pct,
      ndvi_timeline: [{date, ndvi},...],
      landsat_max_ndvi, cloud_cover_pct,
      sar_vv_drop, sar_vh_drop,
      imd_confirmed, imd_disaster_type,
      flags, flag_weights_applied,
      true_color_url, ndvi_map_url, loss_map_url,
      ndvi_timeline_chart_url, sar_image_url,
      model_version, processing_time_ms
    }

GEE DEV FALLBACK (when GEE_ENABLED=false):
  Load mock result from MinIO: /demo/mock_satellite_result.json
  Load demo images from: /demo/true_color.png etc.
  Return immediately with data_source: "MOCK"

PART 8 — CARBON CREDIT ENGINE (NEW MODULE)
════════════════════════════════════════════════════════════════════════

India's Ministry of Agriculture launched the Carbon Credit 
Trading Scheme in Dec 2023. As per government framework, 
farmers implementing regenerative practices can earn credits 
worth USD 40-60 per tonne CO2e with 75% going to farmers.
Grow Indigo has already verified 50,000+ credits via VM0042.

CARBON CREDIT PIPELINE:

STEP 1 — ELIGIBILITY CHECK (run at UDLRN registration):
  CarbonEligibilityService.check(udlrn):
  - Area >= 0.5 ha (minimum viable carbon plot)
  - Land use type = Agricultural
  - Landsat baseline shows green history (ndvi > 0.2)
  - Not blacklisted
  → Set farmers.carbon_eligible = TRUE if passes
  → Show "🌱 Carbon Credits Available" badge on UDLRN card

STEP 2 — FARMER ENROLMENT:
  POST /api/v1/carbon/enrol
  { udlrn, practice_type }
  practice_type options:
    DIRECT_SEEDED_RICE    (DSR - rice planted without flooding)
    NO_TILLAGE            (reduced soil disturbance)
    CROP_RESIDUE          (no stubble burning)
    AGROFORESTRY          (trees on farm boundaries)
    ORGANIC_FARMING       (no synthetic fertilizer)
  
  Creates: carbon_projects record
  Stores: baseline NDVI, baseline carbon estimate
  Methodology assigned (VM0042 for most)
  Monitoring period: 1 year, annual measurement

STEP 3 — SATELLITE MONITORING (every 30 days via @Scheduled):
  For each active carbon_projects:
  Fetch latest Sentinel-2 NDVI for plot polygon
  Compare with baseline NDVI
  Estimate carbon sequestration:
    delta_ndvi = current_ndvi - baseline_ndvi
    sequestration_t_co2e_ha = delta_ndvi * CROP_FACTOR
    total = sequestration_t_co2e_ha * area_ha
  Save: carbon_measurements record
  If delta_ndvi significantly NEGATIVE:
    → FLAG: CARBON_REVERSAL
    → Suspend credit issuance until next measurement

STEP 4 — CREDIT CALCULATION AND ISSUANCE:
  After 12 months of monitoring:
  Total CO2e sequestered = SUM(carbon_measurements)
  Registry validation (VM0042 methodology check)
  Issue carbon_credits record:
    credits_amount = total_co2e
    market_price_usd = current_voluntary_market_price
    farmer_share_pct = 75%
    farmer_payout_inr = credits * price * 0.75 * USD_INR_RATE
  Status: PENDING_VERIFICATION → VERIFIED → SOLD

STEP 5 — CARBON SCORE ON FARMER PROFILE:
  carbon_score = function of:
    practice_duration (months active)
    ndvi_improvement_pct
    consistency (no carbon reversals)
    area enrolled
  Range: 0-100
  Display as "Carbon Health Score" on farmer dashboard
  Linked to UDLRN

PYTHON CARBON ML SERVICE (port 8001):
  POST /carbon/estimate:
  Input: { udlrn, polygon_wkt, practice_type, 
           baseline_ndvi, months_active }
  
  Use NDVI time series to estimate soil organic carbon:
    SOC estimation: empirical model
    soc_tonnes_ha = 0.5 + (ndvi_improvement * 2.3) 
                    * months_active / 12
    # Simplified from VM0042 methodology
    # Full model uses soil type corrections from 
    # SoilGrids dataset (ISRIC)
  
  Return: { estimated_credits, confidence, 
            monthly_estimates, verification_ready }
